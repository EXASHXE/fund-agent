# fund-agent 新闻模块重构设计文档（阶段 1 / 阶段 2）

## 目标

将当前基于 AKShare 泛资讯流的新闻模块，重构为“**基金持仓驱动的定向新闻简报引擎**”。

重构目标：

1. 提高新闻与**持有基金/持仓标的**的相关性。
2. 将新闻从“情绪分数”升级为“**事件简报 + 催化评分**”。
3. 降低重复新闻、无关快讯和宏观噪声对报告质量的干扰。
4. 保持与现有主流程兼容，允许渐进式替换。

---

# 一、重构范围

## 现有问题

当前 news 模块通常存在以下问题：

- 直接抓取全市场新闻，相关性弱。
- 只做粗粒度情绪分析，缺少事件识别。
- 缺少基金持仓映射，无法围绕核心资产采集。
- 缺少去重、过滤、时间衰减、来源权重。
- 输出结果不适合投研简报，更多像新闻列表。

## 目标改造方向

将流程改成：

**基金持仓画像 → 定向新闻采集 → 新闻去重 → 新闻事件蒸馏 → 简报输出**

---

# 二、阶段划分

## 阶段 1：持仓映射 + 定向采集 + 去重

目标：先解决“抓什么新闻”的问题。

### 交付内容

1. 基金持仓画像抽取。
2. 基于持仓的定向新闻采集。
3. 新闻去重与归一化。
4. 形成结构化新闻候选池。

### 输出结果

- `fund_entity_profile`
- `news_candidates`
- `news_deduplicated`

---

## 阶段 2：事件蒸馏 + 时间衰减 + 基础简报

目标：再解决“新闻意味着什么”的问题。

### 交付内容

1. 新闻事件类型识别。
2. 严重度/影响度建模。
3. 时间衰减权重。
4. 输出基金级新闻催化简报。

### 输出结果

- `event_items`
- `weighted_catalyst_score`
- `fund_news_brief`

---

# 三、阶段 1 设计：持仓映射 + 定向采集 + 去重

## 3.1 核心思路

先从基金本身出发，构建“关注实体列表”，再围绕实体去采集新闻，而不是先抓新闻再判断是否相关。

### 核心实体

- 重仓股
- 行业配置
- 资产配置
- 关键主题词
- 基金经理风格相关词（可选）

---

## 3.2 数据输入

### 基金侧输入

来自现有基金分析流程或基金持仓接口，至少包含：

- 基金名称
- 基金代码
- 重仓股列表
- 行业配置
- 持仓更新时间
- 基金风格标签（可选）

### 新闻侧输入

建议优先使用 AKShare 中较适合定向采集的接口：

- 个股新闻
- 个股公告
- 个股研报
- 财经早餐
- 全球快讯
- 财经精选

---

## 3.3 设计流程

```text
基金持仓/行业配置
        ↓
实体抽取与标准化
        ↓
生成新闻关注名单
        ↓
按实体定向采集新闻
        ↓
清洗、归一化、去重
        ↓
输出候选新闻池
```

---

## 3.4 具体模块拆分

建议 `src/news/` 下拆分为以下组件：

```text
src/news/
├── pipeline.py        # 调度入口
├── entity_mapper.py   # 持仓实体构建
├── collector.py       # 统一采集器
├── deduplicator.py    # 去重模块
├── schemas.py         # 数据结构定义
└── sources/
    ├── akshare_source.py
    ├── announcement_source.py
    ├── research_source.py
    └── market_news_source.py
```

## 3.5 Entity Mapper 设计

### 输入

- 基金重仓股
- 基金行业配置
- 基金资产配置

### 输出

```json
{
  "fund_code": "xxxxxx",
  "fund_name": "xxx基金",
  "holdings": [
    {
      "symbol": "600519",
      "name": "贵州茅台",
      "weight": 0.083
    }
  ],
  "sector_keywords": ["白酒", "消费", "高端酒"],
  "theme_keywords": ["提价", "动销", "库存", "渠道"]
}
```

### 规则

- 股票简称、股票代码、英文简称、别名全部标准化。
- 行业词和主题词统一转成白名单。
- 允许手工维护补充别名表。

---

## 3.6 Collector 设计

### 目标

对每个实体去采集相关资讯，而不是抓全网新闻。

### 采集优先级

1. 个股公告
2. 个股新闻
3. 个股研报
4. 行业新闻
5. 宏观背景新闻

### 采集策略

- 以持仓股票为主键拉新闻。
- 对行业关键词做补充检索。
- 同一天多来源新闻合并，但保留来源字段。
- 同一新闻支持多源命中。

---

## 3.7 去重设计

### 去重层级

1. **标题完全重复去重**
2. **内容近似去重**
3. **事件级去重**

### 去重建议

- 先做标题规范化：去空格、去标点、统一大小写。
- 再做内容 embedding 相似度去重。
- 最后按事件实体 + 时间窗口聚合。

### 建议字段

```json
{
  "dedup_key": "600519_2026-05-20_提价",
  "similarity_score": 0.94,
  "source_count": 3
}
```

### Task 2：新增定向采集器

- 重复新闻减少。
- 无关宏观快讯占比下降。
- 任意基金都能生成一份“核心标的相关新闻池”。

# 四、阶段 2 设计：事件蒸馏 + 时间衰减 + 基础简报

## 4.1 核心思路

阶段 1 解决“抓什么”，阶段 2 解决“怎么理解”。

不再把新闻直接映射成单一情绪值，而是提炼成：

- 事件类型
- 影响方向
- 严重度
- 关联实体
- 时间衰减权重
- 最终催化分

---

## 4.2 事件蒸馏设计

### 输入

- 新闻标题
- 新闻正文或摘要
- 新闻来源
- 关联实体
- 发布时间

### 输出

```json
{
  "event_type": "订单下修",
  "polarity": -1,
  "severity": 0.82,
  "impact_scope": "核心持仓",
  "entity": "英伟达",
  "theme": "AI算力链"
}
```

### 事件类型建议

建立一个轻量金融事件字典，至少覆盖：

- 业绩超预期
- 业绩不及预期
- 订单增长
- 订单下修
- CAPEX 扩张
- 去库存
- 涨价
- 降价
- 减产
- 扩产
- 政策利好
- 政策利空
- 风险提示
- 监管调查
- 资产重组

---

## 4.3 事件蒸馏实现方式

可以分两层：

### 第一层：规则/词典优先

适合快速落地，成本低，稳定性高。

- 白名单关键词
- 正负面触发词
- 事件类别映射

### 第二层：LLM 辅助蒸馏

适合复杂文本，尤其是金融、产业、公告类新闻。

- 输入标题 + 摘要 + 原文片段
- 输出标准事件 JSON
- 控制输出格式，避免自由生成

### 建议策略

- 规则优先，LLM 兜底。
- 高价值新闻再调用 LLM。
- 低价值新闻只走规则。

---

## 4.4 时间衰减设计

新闻价值会随时间下降，因此不能简单平均。

### 公式建议

```text
weighted_score = severity × impact × relevance × exp(-λ × Δt)
```

### 参数说明

- `severity`：事件强度，范围建议 `[-1, 1]`。
- `impact`：影响范围，核心持仓权重更高。
- `relevance`：与基金持仓的相关度。
- `Δt`：距离当前时间的天数或小时数。
- `λ`：时间衰减系数。

### 建议默认值

- 日频：`λ = 0.15 ~ 0.30`
- 盘中快讯：`λ` 更高
- 长周期主题：`λ` 更低

---

## 4.5 Relevance 相关度设计

建议相关度由以下部分组成：

- 股票代码命中
- 股票简称命中
- 别名命中
- 行业命中
- 主题词命中
- 事件实体与持仓实体的距离

### 示例

```text
relevance = 0.4 * stock_hit + 0.2 * sector_hit + 0.2 * keyword_hit + 0.2 * alias_hit
```

---

## 4.6 基础简报输出

### 输出目标

不是“新闻列表”，而是“基金新闻简报”。

### 建议输出结构

```json
{
  "fund_code": "xxxxxx",
  "date": "2026-05-21",
  "weighted_catalyst_score": -0.37,
  "trend": "bearish",
  "top_events": [
    {
      "title": "某公司订单下修",
      "entity": "某公司",
      "event_type": "订单下修",
      "score": -0.91
    }
  ],
  "sector_summary": {
    "AI算力链": -0.63,
    "消费": 0.12
  }
}
```

## 4.7 阶段 2 的代码重构任务清单

### Task 1：新增事件蒸馏器

- 输入新闻文本。
- 输出标准事件 JSON。
- 支持规则和 LLM 两种路径。

### Task 2：新增权重计算器

- 实现 severity、impact、relevance、time decay。
- 输出单条新闻催化分。

### Task 3：新增聚合器

- 将多条新闻聚成基金级舆情结果。
- 生成 top events 和 sector summary。

### Task 4：替换旧的情绪平均逻辑

- 不再只输出 SnowNLP 风格分数。
- 保留兼容字段，但主输出改为事件级简报。

---

## 4.8 阶段 2 验收标准

- 简报能指出“发生了什么事件”。
- 简报能区分核心持仓和外围噪声。
- 同一天多条新闻不会简单重复堆叠。
- 输出具备明显投研价值，而不是情绪均值。

# 七、给 Code Agent 的执行流程

建议按以下顺序执行：

## 第 1 步：梳理现有接口

- 找出现有新闻入口文件。
- 找出当前调用的 AkShare 接口。
- 记录当前输出结构。

## 第 2 步：新增实体画像层

- 从基金持仓和行业配置生成 entity profile。
- 为后续采集提供统一输入。

## 第 3 步：重写采集层

- 基于实体 profile 调用多源采集。
- 输出统一新闻 schema。

## 第 4 步：增加去重层

- 实现标题去重、语义去重、事件归并。

## 第 5 步：重写分析层

- 新增事件蒸馏。
- 新增 time decay。
- 新增 catalyst score。

## 第 6 步：保留兼容出口

- 旧字段可保留一段时间。
- 新字段成为主输出。

## 第 7 步：补测试

- 单测覆盖采集、去重、事件识别、衰减、聚合。

# 八、建议的测试用例

## 阶段 1 测试

1. 基金持仓为空。
2. 单只股票有多条重复新闻。
3. 同一新闻来自多个来源。
4. 行业关键词命中但股票代码未命中。

## 阶段 2 测试

1. 利好事件和利空事件混合。
2. 旧新闻和新新闻同时存在。
3. 核心持仓新闻和外围新闻同时存在。
4. 公告类文本与普通新闻文本混合。

# 十、最终落地效果

完成阶段 1 和阶段 2 后，新闻模块的输出会从：

- 低相关、重复多、噪声大的新闻流

升级成：

- 围绕基金重仓资产的定向新闻池
- 具备事件分类与催化判断能力
- 带时间衰减和相关度权重
- 可以直接给 LLM 写投研简报

这会让整个 fund-agent 的报告质量明显提升。
