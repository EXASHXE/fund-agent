# v0.9.0 Release Commands

> Step-by-step instructions for finalizing and tagging the v0.9.0 release.
> Do NOT execute these commands unless explicitly instructed.

## 1. Verify clean working tree

```bash
git status --short
```

There should be no uncommitted changes. If there are, commit or stash them first.

## 2. Run final tests

```bash
PYTHONPATH=. pytest -q
fund-agent regressions --pretty
fund-agent doctor --pretty
python scripts/smoke_host_install.py
```

All tests must pass before proceeding.

## 3. Create annotated tag

```bash
git tag -a v0.9.0 -m "fund-agent v0.9.0 pre-launch baseline"
```

## 4. Push tag

```bash
git push origin v0.9.0
```

## 5. Verify tag

```bash
git tag -l v0.9.0
git show v0.9.0 --stat
```

## 6. Compare v0.9.0 to master

```bash
git log v0.9.0 --oneline -10
git diff v0.9.0..HEAD --stat
```

## 7. Optional GitHub release

```bash
gh release create v0.9.0 --title "v0.9.0" --notes "fund-agent v0.9.0 pre-launch baseline"
```

## 8. Recovery if tag was created incorrectly

If the tag was created but not yet pushed:

```bash
git tag -d v0.9.0
```

If the tag was already pushed:

```bash
git push origin :refs/tags/v0.9.0
git tag -d v0.9.0
# Then re-create the tag after fixing the issue
```

## Optional clean-repo snapshot strategy

- Preserve existing repository history.
- Treat v0.9.0 as the first pre-launch baseline.
- Earlier commits are experimental development history.
- If a clean public repo is wanted, initialize a new repository from the v0.9.0
  snapshot instead of rewriting this repository.
- Do not force-push or rewrite current public history unless there is a security
  incident.

Only recommend history rewrite if:

- Secrets were committed
- Private data was committed
- Legal/security issue requires rewrite

Do not perform history rewrite.
